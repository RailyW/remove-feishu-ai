# `internal/app`

## 模块职责

`internal/app` 模块负责把整个控制台程序真正“跑起来”。

它不实现底层搜索算法，也不直接定义补丁规则；它的职责是把已经存在的能力按产品要求串成一条稳定的执行链：

1. 启动即检查管理员权限，不足时通过 UAC 重启。
2. 在 exe 同目录读取 `config.json`，解析 `backup` 目录。
3. 每次启动都要求用户确认飞书安装路径，但允许直接回车复用上次路径。
4. 扫描两个用户可见功能当前状态，并把扫描缓存写回配置。
5. 构建主菜单，根据状态决定哪些动作可用。
6. 在用户选择动作后创建事务，执行补丁、提交或回滚。
7. 提供查看最近备份、重新选择路径等辅助流程。

## 文件说明

### [`app.go`](./app.go)

本模块的主编排文件，包含以下核心内容：

- `App`：应用入口依赖容器，持有 logger、console、配置读写、路径解析、事务工厂和默认目标功能集合。
- `featureTarget`：描述一个用户可见功能，以及它由哪些底层补丁组件共同组成。
- `FeatureSnapshot`：描述一次扫描后，菜单真正使用的用户可见状态。
- `Run()`：程序主循环，负责权限检查、启动参数、环境初始化、状态扫描、菜单交互和动作执行。
- `bootstrap()`：读取配置、解析程序目录、确认安装路径。
- `scan()`：检测两个功能状态，并把偏移缓存与 JS bundle 路径缓存写回配置。
- `aggregateStates()`：把多个底层组件状态合成为一个用户可见状态。
- `buildMenuItems()`：根据状态决定菜单项是否可用。
- `executeChoice()`：把菜单动作分发到批量禁用、批量恢复、单项禁用、单项恢复、重选路径或查看最近备份。

### [`env.go`](./env.go)

定义整个应用运行过程中共享的环境对象：

- `Env`：保存 `config.json` 路径、exe 目录、飞书安装目录、备份根目录、严格模式开关、当前配置和 logger。
- `InstallPath()`：满足底层 `feature.Env` 接口，让补丁模块只依赖“安装目录”这一最小契约。
- `ResolveBackupRoot()`：把配置中的 `backup_root` 解析为真正的落盘目录，确保默认落在程序目录下的 `backup\`。

### [`app_test.go`](./app_test.go)

覆盖本模块的关键行为：

- 菜单项是否正确反映当前状态。
- 多个底层组件如何合成为一个用户可见状态。
- 严格模式下批量动作遇到失败时是否触发事务回滚。

## 关键设计

### 用户只看到两个功能，不直接看到底层补丁点

虽然当前实现底层上包含：

- `frame.dll` 规则
- 知识问答侧边栏 JS 规则
- 群聊总结 JS 规则

但菜单层只展示两个用户可见功能：

- 侧边栏知识问答
- 群聊 AI 消息速览/群聊总结

其中“侧边栏知识问答”会聚合 `frame.dll` 和知识问答 JS 两个底层组件的状态与动作，不把实现细节直接暴露给终端用户。

### 组合状态规则保持保守

当一个用户可见功能由多个底层组件组成时：

- 全部 `original` 才视为 `original`
- 全部 `patched` 才视为 `patched`
- 同时出现 `original` 与 `patched`，或任一组件本身就是 `mixed`，则整体视为 `mixed`
- 其余情况统一视为 `unknown`

这样可以避免只改了一部分组件却仍然把整个功能误判成“已完全禁用”或“已完全恢复”。

### 严格模式与非严格模式的事务边界不同

批量操作下：

- 严格模式：整个批量动作共用一份事务；任一目标失败，就回滚本次批量中已经成功的所有修改。
- 非严格模式：每个用户可见功能各自使用独立事务；单个功能失败时只回滚它自己的组件，不影响其他已经成功提交的功能。

这样既保留了默认严格模式的一致性，也避免了非严格模式下出现“同一个功能的两个底层组件只成功了一半”的问题。

### 缓存只做加速，不改变安全边界

`scan()` 会把：

- `frame.dll` 的偏移缓存
- 最近命中的 JS bundle 相对路径
- JS 命中文件的文件大小与修改时间

写回 `config.json`，但这些缓存只用于加速下次定位。真正是否允许写入，仍然由每个底层模块自己的严格状态检测决定。

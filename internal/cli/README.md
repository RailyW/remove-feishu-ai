# `internal/cli`

## 模块职责

`internal/cli` 模块只负责“控制台如何和用户说话”。

它不做文件扫描，不做补丁，不做事务，也不理解飞书版本兼容性。它只接收应用层已经整理好的信息，并完成两类交互：

1. 路径确认：提示用户输入飞书安装路径，允许直接回车采用上次值。
2. 菜单交互：展示安装路径、严格模式、功能状态和可选操作，并把用户输入的编号转换成具体动作。

## 文件说明

### [`menu.go`](./menu.go)

定义控制台菜单相关的数据结构与主菜单交互流程：

- `Action`：菜单动作枚举，例如批量禁用、单项恢复、重选路径、查看备份、退出。
- `MenuItem`：描述单个菜单项的文案与当前是否可用。
- `StatusLine`：描述一个用户可见功能当前状态。
- `MenuScreen`：描述一次完整主界面需要展示的所有信息。
- `Console`：封装输入输出流与菜单选择能力。
- `SelectAction()`：渲染菜单并循环读取合法编号。

### [`prompt.go`](./prompt.go)

定义更基础的提示输入能力：

- `PromptInstallPath()`：提示用户输入安装路径，空输入时回退到当前路径。
- `readLine()`：安全读取一行文本，兼容没有换行符的 EOF 输入。

### [`menu_test.go`](./menu_test.go)

覆盖“菜单项被禁用时，Console 不应直接接受该编号，而是继续提示”的行为。

### [`prompt_test.go`](./prompt_test.go)

覆盖“用户直接回车时，安装路径提示应当返回当前默认路径”的行为。

## 关键约束

### CLI 只处理交互，不理解底层业务

`internal/cli` 接触到的对象只有：

- 展示文本
- 状态文本
- 菜单动作 ID

它不知道 `frame.dll` 在哪里，也不知道 JS bundle 是什么，更不知道事务怎么回滚。这样可以保证交互层足够稳定，后续即使底层补丁逻辑变化，也不需要重写整套终端 UI。

### 输出必须克制

本模块不会打印任何大文件内容，也不会把最小化 JS 或二进制字节转成超长日志输出到终端。它只显示：

- 当前路径
- 严格模式说明
- 两个功能状态
- 菜单项文案

### 禁用项仍然显示

菜单中即使某个动作当前不可执行，也不会直接隐藏，而是以“当前不可用”的形式继续展示。这样用户能明确看到程序支持哪些能力，以及为什么当前状态下不能执行某一步。
